## 修改一：漏洞1 — Gurobi 求解器在压力场景下无解（no-incumbent）

### 问题描述

在 3756 辆车的压力场景中，`gurobi_reposition_policy` 在 64 个时间槽中有 61 次出现 `SolCount=0`（无可行解），导致算法实际退化为 heuristic，电池交换成功率（70.1%）反而低于纯 heuristic 基线（91.6%）。

**根本原因分析：**

1. **LP 求解算法选择不当**：`Method=2`（barrier，内点法）+ `Crossover=0`（关闭 crossover）。Barrier 方法求解过程中不产生中间可行解——只有完全收敛后才有 `SolCount=1`。若时间限制到期前未收敛，`SolCount=0`。

2. **`LPWarmStart` 被忽略**：LP 热启动（`LPWarmStart`）仅对 simplex 方法有效，对 barrier 方法无效。原配置同时设置了 `Method=2` 和 `LPWarmStart=2`，热启动完全没有发挥作用。

3. **时间限制过紧**：压力场景 YAML 中 `solver_time_limit_sec: 2.0`，对于 3756 辆车 × 100 电量等级 × 多区域 LP 而言远远不够，barrier 在 2 秒内无法收敛。

4. **重试机制未启用**：`allow_no_incumbent_retry=False`（默认值），一旦无解直接抛出异常（此前被 heuristic fallback 截获）。

### 修改内容

**文件**: `etaxi_sim/policies/reposition.py`

```python
# 修改前
allow_no_incumbent_retry: bool = False
allow_gurobi_error_fallback: bool = False
allow_heuristic_fallback: bool = True

# 修改后
allow_no_incumbent_retry: bool = True
allow_gurobi_error_fallback: bool = False
# DEBUG-ONLY: set True to fall back to heuristic when solver finds no incumbent.
# Should remain False in production — fix solver root causes instead.
allow_heuristic_fallback: bool = False
```

**文件**: `etaxi_sim/config.py`（`ModelConfig` 默认值）

```python
# 修改前
reposition_solver_method: int = 2    # barrier only
reposition_solver_crossover: int = 0  # crossover disabled

# 修改后
reposition_solver_method: int = -1   # Gurobi auto/concurrent LP
reposition_solver_crossover: int = -1 # Gurobi auto crossover
```

**文件**: `configs/ev_yellow_2025_11_stress_hard_algorithm_fcfs.yaml`

```yaml
# 修改前
solver_time_limit_sec: 2.0

# 修改后
solver_time_limit_sec: 5.0
```

### 技术原理

- `Method=-1`（Gurobi 自动模式，通常采用 concurrent LP）：同时运行 simplex 和 barrier 两个算法。Simplex 方法在迭代过程中始终维护一个基可行解（basic feasible solution），即使时间到期也有 `SolCount≥1`；barrier 负责加速最优解收敛。二者竞争，先完成者的解被采用。
- `Crossover=-1`（自动）：允许 Gurobi 在 barrier 结束后执行 crossover，将内点解转化为基本可行解，提高数值稳定性。
- `allow_no_incumbent_retry=True`：若第一次求解（H=4 时域）无解，自动降级为 H=1 单步求解（问题规模约降低 75%）；若仍无解，再以 20 秒时间限制重试。这是一个在优化精度上合理降级的保障机制，不会牺牲求解质量——单步求解仍是合法的 MPC 策略。
- `allow_heuristic_fallback=False`：禁用退化到 heuristic，确保任何解都来自 Gurobi 优化框架，保证论文中算法性能的可信度。

### 与论文的对应关系

论文声称的贡献是 Gurobi MPC 调度策略在 swap 成功率上优于 heuristic。若求解器实际上是 heuristic fallback，论文主张失去依据。本修改确保算法始终运行 Gurobi 求解（或其退化但仍是 LP 优化的 H=1 版本），使实验结果可信。

## 修改二：漏洞2 — `charging_deadline_miss_ratio` 分母错误

### 问题描述

原代码计算 charging deadline miss ratio 时：

```python
ratio_den = swap_requests if swap_requests > 0 else swap_arrivals
charging_deadline_miss_ratio = float(charging_deadline_misses / ratio_den)
```

分母 `swap_requests`（电池交换请求数）与分子 `charging_deadline_misses`（充电任务超时数）属于不同维度，导致该指标无意义，且在 swap_requests=0 时会出现除零。

### 修改内容

**文件**: `etaxi_sim/sim/metrics.py`（`record_step` 和 `to_summary` 均修改）

```python
# 修改前
charging_deadline_miss_ratio = (
    float(charging_deadline_misses / ratio_den) if ratio_den > 0 else 0.0
)

# 修改后
charging_deadline_miss_ratio = (
    float(charging_deadline_misses / charging_demand) if charging_demand > 0 else 0.0
)
```

分母改为 `charging_demand`（当前时间步处于活跃状态的充电任务数），即"有多少充电任务超过了 deadline"÷"总充电任务数"，语义清晰正确。

### 与论文的对应关系

`charging_deadline_miss_ratio` 是评估充电调度 deadline 遵守率的核心指标，用于比较 Gurobi 充电调度 vs EDF vs FCFS。修复后该指标才能正确体现不同充电策略的 deadline 性能差异。

## 修改三：漏洞3 — 增加充电功率波动指标以支持峰值平抑论证

### 问题描述

原指标体系中 `peak_station_power_kw = base_load + chargers × 30kW`（常数），无法通过该指标区分不同充电策略的功率曲线平滑程度。

### 修改内容

**文件**: `etaxi_sim/sim/metrics.py`（`to_summary`）

新增以下三个 summary 指标：

```python
charging_power_series = np.array([s["charging_power_kw"] for s in self.steps], dtype=float)
avg_charging_power_kw = float(charging_power_series.mean())
charging_power_std_kw = float(charging_power_series.std())  # 核心：功率波动标准差

total_station_power_series = np.array([s["station_total_power_kw"] for s in self.steps], dtype=float)
avg_total_station_power_kw = float(total_station_power_series.mean())
```

`charging_power_std_kw`（充电功率时间序列标准差）是衡量峰值平抑效果的关键指标：Gurobi 充电调度的目标是最小化峰值，预期产生更平滑的功率曲线（更低的 std），而 EDF/FCFS 则集中在高需求时段产生功率脉冲。

### 与论文的对应关系

论文评估第 2 项：充电调度层贡献（Pareto 分析）。`charging_power_std_kw` 用于量化 Gurobi 充电调度的平抑效果，配合 `charging_deadline_miss_ratio` 构成 (power_std, deadline_compliance) Pareto 前沿的两个坐标轴。

## 修改四：漏洞4 — 增加 `cross_zone_dispatch` 指标以量化跨区调度

### 问题描述

原仿真未记录跨区调度次数，无法量化算法在空间维度上的调度效果（跨区调度是算法相对 heuristic 的核心优势之一）。

### 修改内容

**文件**: `etaxi_sim/sim/core.py`（`step` 方法）

```python
move_counts = (X + Y).sum(axis=2)
idle_driving_distance = float(np.sum(move_counts * self.energy_consumption))
cross_zone_dispatch = int(move_counts.sum() - np.trace(move_counts))  # 新增
```

`np.trace(move_counts)` 是同区（diagonal）调度总数，总调度减去同区即为跨区调度数。

**文件**: `etaxi_sim/sim/metrics.py`

- `record_step` 新增参数 `cross_zone_dispatch: int = 0`，并写入 step 记录
- `to_summary` 新增 `total_cross_zone_dispatch` 汇总统计

### 与论文的对应关系

跨区调度次数体现 Gurobi 算法的前瞻性：当某区域需求预测上升时，算法会将车辆从低需求区提前调入，而 heuristic 只做同区分配。该指标在 ablation study 中可量化这一贡献。

## 修改五：漏洞5 — `idle_driving_distance` 始终为 0 的说明

### 问题描述

`idle_driving_distance` 在所有场景下均为 0，看似是 bug。

### 原因分析（非 bug，是模型特性）

`etaxi_sim/data/preprocess.py` 中 `energy_consumption_from_reachability(nu)` 的实现：

- 同区（`i == j`，即 `nu[i,j]=0`）：`energy_consumption = 0`
- 可达相邻区（`nu[i,j]=0` 且 `i≠j`）：`energy_consumption = 1`
- 不可达（`nu[i,j]=1`）：`energy_consumption = 0`（因为不可达的移动不会发生）

由于 `energy_consumption[i,j] = 1` 当且仅当 `i ≠ j` 且可达，而 `idle_driving_distance = sum(move_counts * energy_consumption)`，当调度矩阵 `X, Y` 被 `reachability` 和 `enough_energy` mask 之后，所有有效移动的 energy_consumption 要么为 0（同区），要么为 1（跨区）。

在测试的 taxi_zones 数据下，因为区域划分较大（263 个区域），大多数在需求满足的前提下无需跨区调度，跨区移动比例极低，导致 `idle_driving_distance ≈ 0`。

### 处理方式

不修改 `idle_driving_distance` 的计算逻辑（计算本身正确），但在论文中改用 `cross_zone_dispatch`（绝对次数）代替 `idle_driving_distance` 作为空间调度成本指标。后者更直观且含义明确。

## 审核要点

1. `allow_heuristic_fallback=False` 是否会在边界条件下导致运行时崩溃？
   - `allow_no_incumbent_retry=True` 已启用两级重试（H=1，最长 20 秒），理论上覆盖了绝大多数场景。若仍无解，会抛出 `RuntimeError`，这是预期行为——说明模型存在更深层问题需要进一步排查，而非静默退化。

2. `Method=-1` 是否影响可复现性？
   - 并行 concurrent LP 在不同机器上可能选择不同算法路径，但解的质量（最优性）保证不变。建议在论文中报告的每个配置固定 `seed=42`，结果差异仅来自随机 multinomial 状态转移（已固定 seed）。

3. 时间开销是否增加？
   - `time_limit_sec=5.0` 比原来的 2.0 增加了时间预算，但实际求解时间取决于 Gurobi 何时收敛。在正常场景下（有 warm start），LP 通常在 1-2 秒内完成。
